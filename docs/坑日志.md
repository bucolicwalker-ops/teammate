### [2026-06-24] 配模型：qwen 全 403，最后发现是端点/协议选错了
- **现象**：用 OpenAI SDK + `compatible-mode` 端点调 qwen-plus/turbo/max，全部 403 `Model.AccessDenied`。
- **根因**：这把 DashScope key 是配在 **Anthropic 兼容端点**（`/apps/anthropic`，serve 智谱 GLM）上的，不是 OpenAI 兼容端点（serve qwen）。端点 + SDK 协议都选错了。
- **定位过程**（关键是分层排除，不是瞎试）：
  1. 看错误码——是 **403(权限)** 不是 **401(key 错)** → key 有效，是权限问题。
  2. 探多个 qwen 模型**全 403** → 排除"单模型没开通"，指向账号/端点层面。
  3. 关键线索：本地 Claude Code 能用这把 key → CC 走的是 **Anthropic 协议** → 换 anthropic SDK + `/apps/anthropic` 端点。
  4. 探 GLM 模型名，确认 `glm-5.1`（带横杠）可用；`glm5.1` 报 InvalidParameter。
- **解决**：换 anthropic SDK，`base_url=.../apps/anthropic`，`model=glm-5.1`。
- **结果**：一次跑通，MyAgent 正常应答。
- **总结**：调模型报 403，我先用错误码区分"key 错(401)还是权限错(403)"，再用多模型探测把"单模型"和"全账号"分开，最后从"另一个工具能用同一把 key"反推出是**端点/协议不匹配**——换对端点和 SDK 解决。定位靠分层排除，不是穷举瞎试。

### [2026-06-24] MyAgent「失忆」：原来 LLM 调用本身是无状态的
- **现象**：跟 MyAgent 说「我叫小明」，下一句问「我叫什么」，它完全不记得。
- **根因**：每次 `client.messages.create()` 都是一次**独立、无状态**的请求；模型不会自动记住上一轮。`ask()` 每次只把"当前这句"发过去，没带历史。
- **定位过程**：翻 `agent.py` 的 `ask()`——`messages` 数组每次只有当前 user 消息、没有累积历史 → 确认是"我们没喂历史"，不是模型坏了。
- **解决方向**（W3 要做的）：① **短期记忆** = 把对话历史一起塞进 `messages`（受上下文窗口限制）；② **长期记忆** = 把要点存向量库，需要时检索回来（RAG）。
- **总结**：LLM API 是无状态的——「记忆」不是模型自带的，是工程上自己维护、再喂回去的。短期靠传 history（受上下文窗口约束），长期靠外部存储 + 检索。这也正是「上下文窗口」和「记忆」成为常见问题的原因。

### [2026-06-25] 加了字段却没同步 prompt：重试 3 次都救不回来
- **现象**：给 `AgentReply` 加了 `urgency` 字段，但 `SYSTEM` 没提它。运行后模型一直不输出 urgency，校验失败，**重试 3 次全挂**，模型死活不补。
- **根因**：**两个「真相源」脱节**——`SYSTEM`（告诉模型"字段只有 intent/reply/need_followup"）vs `AgentReply` 模型（校验时要 4 个字段）。模型按 SYSTEM 办事，和校验器**直接矛盾**；重试只是把同一个矛盾重复了 3 遍。（雪上加霜：重试话术把"缺字段"说成"无法解析、请返回合法 JSON"，可模型返回的本就是合法 JSON，更没法纠正。）
- **定位过程**：观察到 3 次错误一模一样、模型完全没尝试加 urgency → 意识到不是"模型笨"，是它收到的指令**自相矛盾**。
- **解决**：把 `SYSTEM` 改成 `+ json.dumps(AgentReply.model_json_schema())`——字段说明**从模型自动生成**。契约只剩一处（`AgentReply` 类），SYSTEM 自动带上 urgency → 模型第一次就给 → 一次过、零重试。
- **总结**：重试 / 错误反馈**救不了"prompt 描述和 schema 从根上矛盾"**——打补丁打不过源头矛盾。真正的修法是**单一真相源**：让 prompt 从 schema 自动生成，二者永不脱节。我一开始靠重试兜底，后来定位到根因是契约漂移，改成单一真相源才彻底解决。

### [2026-07-03] 截断破坏 user/assistant 配对——history 切片砍到 turn 中间 API 400
- **现象**：`_truncate()` 盲切片 `self.history[-max_history:]`，history 超过 20 条时 API 报 400——Anthropic 要求 messages 严格交替 user/assistant，切片后首条可能是 assistant 或 tool_result。
- **根因**：工具调用产生 4 条消息 `[user, assistant(tool_use), user(tool_result), assistant(text)]`，盲切片从一个 turn 中间断开 → 首条不是 proper user 消息 → API 拒绝。
- **定位过程**：code review 时推演了截断点落在各种位置的情况——截断落在 assistant → pop → 下一条 tool_result content 是 list 不是 str → pop → 链式清到 proper user。happy path 永远碰不到（测试用例不到 20 条消息）。
- **解决**：切片后从头部 `while` pop 不完整消息，直到首条是 `role=user + content=str`。利用"合法序列碎片只在头部"这个不变量链式清理。
- **结果**：`max_history=4` 跑 5 轮不崩（之前一定崩）。
- **总结**：截断 history 我用了盲切片，结果 API 报 400——Anthropic 要求 messages 严格交替，切片从一个 turn 中间断开就破坏配对。修法是切片后从头部清理不完整的消息直到首条是 proper user。关键洞察是"合法序列碎片只在头部"——因为只砍头部保留尾部。这个 bug happy path 永远碰不到，只有真实对话超 20 条才触发。

### [2026-07-09] eval judge 摸鱼——faithfulness 0.55 是假数据
- **现象**：RAG Triad baseline 跑出来 faithfulness 0.55，看似是 grounding gap。但读 `eval_baseline.json` 发现 9/10 题 faithfulness 都是 0.50 + unsupported claims 全空 `[]`。
- **根因**：judge prompt 要求"列出所有 claims"→ 答案有表格+多段，judge 生成很长的 JSON → `max_tokens` 截断 → JSON 不完整 → parse 失败 → fallback `{"score": 0.5}`。
- **定位过程**：review baseline JSON 发现"数字太一致=可疑"——9/10 都是精确的 0.50 且 unsupported 全空。不是"LLM 编造"，是 judge 根本没认真判。
- **解决**：简化 prompt——不列 all claims，只列 unsupported + score。响应短了不截断，judge 认真判了。重跑 baseline：faithfulness 0.89（9/10 及格）。
- **结果**：0.55（假）→ 0.89（真）。F-Q12 faithfulness 0.50 是唯一真 gap（LLM 加了知识库里没有的 AI 搜索知识）。
- **总结**：我的 RAG 评估第一版 faithfulness 0.55 看起来是 grounding 问题，但 review 时发现 9/10 题都是 0.50 且 unsupported 全空——judge 在摸鱼。根因是 prompt 要求列所有 claims 导致响应过长被 max_tokens 截断，JSON 解析失败走了 fallback 0.50。简化 prompt 后真实分数 0.89。教训是：eval 框架本身也要被验证，不能假设 judge 一定认真。

### [2026-07-11] _route 子串误匹配——"不需要 researcher" 误路由到 researcher
- **现象**：Supervisor 多 Agent 路由测试中，LLM 返回 "不需要 researcher 用 writer"，但 `_route()` 用 `if role in decision` 子串匹配 → "researcher" 在 "不需要 researcher" 里匹配到了 → 误路由到 researcher。
- **根因**：`"researcher" in "不需要 researcher 用 writer"` → True（子串匹配），先匹配到 researcher 就返回了。
- **解决**：取首词 `decision.split()[0]` + 精确匹配 `decision == role`。如果 LLM 返回 "不需要" → 不匹配任何 role → 返回 done（安全降级）。
- **总结**：Supervisor 路由用 `if role in decision` 子串匹配，LLM 返回"不需要 researcher"时误匹配到 researcher。改成取首词加精确匹配。路由错了整个多 Agent 流乱——这个 bug 说明字符串匹配要精确不能子串。

### [2026-07-14] OSError 子类 bug——FileNotFoundError 被当临时性错误重试了
- **现象**：`read_file('nonexistent.txt')` 报错后，`execute_tool` 重试了 2 次——FileNotFoundError 是永久性错误，重试不会让文件凭空出现。
- **根因**：`RETRYABLE_ERRORS = (TimeoutError, ConnectionError, OSError)` → FileNotFoundError 是 OSError 子类 → 匹配 RETRYABLE → 被重试。应该只重试真正临时性的错误。
- **定位过程**：接了 `read_file` 真工具后撞到——mock 工具（get_weather/search）永远不会抛异常，所以这个分类 bug 一直没暴露。真实用驱动的直接产物。
- **解决**：去掉 OSError，只留 `(TimeoutError, ConnectionError)`。FileNotFoundError/PermissionError 落到 `except Exception` → 永久性错误不重试，直接错误回灌给模型。
- **总结**：我把 OSError 放进重试列表，结果 FileNotFoundError 被重试了——因为 FileNotFoundError 是 OSError 子类，重试不会让文件凭空出现。修法是只放 TimeoutError 和 ConnectionError。这个 bug mock 工具永远不会触发——只有接了真工具 read_file 才暴露，这就是真实用驱动的价值。

### [2026-07-14] MCP notification 流串行——server 回了响应 client 没读导致错位
- **现象**：MCP 测试中 `read_file` 返回 "Unknown method: notifications/initialized"，`calc` 返回了 `read_file` 的结果——响应流错位了一轮。
- **根因**：JSON-RPC 2.0 规范——notification（无 id 的请求）不应该回响应。但 mcp_server 对所有请求都回了响应（包括 notification）。client 对 notification 不读响应（按规范）→ server 的响应积在 stdout buffer → 下一次请求读到了上一轮的 notification 响应 → 错位。
- **解决**：mcp_server 的 main 循环加判断——`if "id" not in req: continue`（notification 静默处理，不回响应）。
- **总结**：MCP 协议踩了 JSON-RPC 2.0 的坑——notification（无 id）不应该回响应，但我的 server 对所有请求都回了。client 不读 notification 响应导致 buffer 积压，后续请求读到了上一轮的响应造成错位。按规范修了 server 不回 notification 响应。踩了这个坑才理解为什么 JSON-RPC 2.0 要区分 request（有 id 回响应）和 notification（无 id 不回）。

### [2026-07-14] read_file OOM——`f.read()[:2000]` 先读整个 1GB 文件再截断
- **现象**：`read_file` 的 `f.read()[:2000]` 看起来只返回 2000 字，但实际上先 `f.read()` 读整个文件到内存再切片——1GB 文件直接 OOM。
- **根因**：`f.read()` 读全部 → `[:2000]` 切片。切片发生在读完之后，大文件已经 OOM 了。
- **解决**：`f.read(2000)`——只读前 2000 字，不读整个文件。一字之差，OOM vs 不 OOM。
- **总结**：read_file 我写的是 f.read()[:2000]——看起来只返回 2000 字但实际先读整个文件再截断。1GB 文件直接 OOM。改成 f.read(2000) 只读前 2000 字。一字之差，根因是 Python 的 read() 不带参数读全部，带参数只读 N 字。

### [2026-07-15] SIGALRM 只主线程——服务化后 uvicorn 线程池里 signal() 报 ValueError
- **现象**：`python -m src.server` 起服务后 curl 请求，`calc` 工具返回"计算工具出了点小问题"。但 `python -m src.agent` 脚本模式跑得好好的。
- **根因**：`_call_with_timeout` 用 `signal.SIGALRM` 实现超时——SIGALRM 只在主线程有效。uvicorn 处理请求时跑在线程池（子线程）→ `signal.signal(SIGALRM)` 抛 `ValueError: signal only works in main thread` → 被 `except Exception` 捕获 → 错误回灌给 LLM → LLM 说"工具出了问题"。
- **定位过程**：脚本模式（主线程）正常，服务模式（线程池）报错——差异在线程。确认是 SIGALRM 的线程限制。
- **解决**：换成 `concurrent.futures.ThreadPoolExecutor`——跨线程安全，在 uvicorn 线程池里也能用。
- **总结**：我用 SIGALRM 实现工具超时控制，脚本测试通过。但服务化后 uvicorn 用线程池处理请求，子线程里 signal() 报 ValueError——SIGALRM 只在主线程有效。换成 ThreadPoolExecutor 修复。这个 bug 只有服务化才暴露——脚本模式永远跑在主线程。

### [2026-07-15] 服务化并发串记忆——多用户共享一个 agent 实例
- **现象**：W8 服务化后所有请求共享一个 MyAgent 实例——用户 A 说"我叫小明"，用户 B 问"我叫什么"→ recall "小明"（串了）。
- **根因**：server.py 创建一个 agent 实例在模块级——所有请求共用 history 和 memory。
- **解决**：per-user agent 实例字典 `agents = {}`，每个用户独立 memory 文件 `data/memory_{user_id}.json` + 独立 history。知识库只读共享（只加载一次）。加 `threading.Lock` 串行化 `ask()` 防并发。加 LRU 淘汰（MAX_AGENTS=50）防内存泄漏。
- **总结**：服务化后所有用户共享一个 agent 实例导致记忆串了。解法是 per-user agent 实例——每个用户独立的 memory 文件和 history，知识库只读共享只加载一次。还要加 threading.Lock 串行化防并发 + LRU 淘汰防 1000 用户 1000 子进程泄漏。

### [2026-07-15] Docker 路径 bug——本地 work Docker 不 work
- **现象**：`docker-compose up` 后 server 启动，但 KB 加载 FileNotFoundError——`../../bagu/qbank.md` 在容器里解析成 `/bagu`（src/../../bagu），但 volume 挂的是 `/app/bagu`。
- **根因**：`os.path.join(os.path.dirname(__file__), "..", "..", "bagu")` 是相对源码文件路径——本地（`src/server.py` → `../../bagu` 正确）和 Docker（`/app/src/server.py` → `/app/src/../../bagu` = `/bagu` 错误）解析不同。
- **解决**：用 `KB_PATH` 环境变量——本地默认用相对路径，docker-compose 设 `KB_PATH=/app/bagu/qbank.md`。
- **总结**：KB 路径用了相对源码文件的 `../../bagu`——本地 work 但 Docker 容器里 `src/../../bagu` 解析成 `/bagu` 不是 `/app/bagu`。修法是加 KB_PATH 环境变量，docker-compose 里设成 `/app/bagu/qbank.md`。教训是路径不要用相对源码的 `..`——要能用环境变量覆盖。

### [2026-07-15] agents 字典无限增长——1000 用户 = 1000 agent + 1000 子进程泄漏
- **现象**：每个新 user_id 创建一个 MyAgent + MCPClient 子进程，永不释放。1000 用户 = 1000 agent 实例 + 1000 个 mcp_server 子进程 → 内存泄漏 + 进程泄漏。
- **根因**：`agents = {}` 只增不减，没有淘汰机制。
- **解决**：`MAX_AGENTS=50` LRU 淘汰——超过上限时销毁最旧的 agent，调用 `old.mcp_client.close()` 终止子进程。`threading.Lock` 保护淘汰逻辑防并发竞态。
- **总结**：per-user agent 每个新用户创建一个实例加子进程，没有淘汰机制——1000 用户就 1000 子进程泄漏。加了 LRU 淘汰 MAX_AGENTS=50，超限时销毁最旧的 agent 并 close 子进程。生产级必须有淘汰机制——不能只增不减。
