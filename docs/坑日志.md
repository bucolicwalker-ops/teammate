### [2026-06-24] 配模型：qwen 全 403，最后发现是端点/协议选错了
- **现象**：用 OpenAI SDK + `compatible-mode` 端点调 qwen-plus/turbo/max，全部 403 `Model.AccessDenied`。
- **根因**：这把 DashScope key 是配在 **Anthropic 兼容端点**（`/apps/anthropic`，serve 智谱 GLM）上的，不是 OpenAI 兼容端点（serve qwen）。端点 + SDK 协议都选错了。
- **定位过程**（关键是分层排除，不是瞎试）：
  1. 看错误码——是 **403(权限)** 不是 **401(key 错)** → key 有效，是权限问题。
  2. 探多个 qwen 模型**全 403** → 排除"单模型没开通"，指向账号/端点层面。
  3. 关键线索：本地 Claude Code 能用这把 key → CC 走的是 **Anthropic 协议** → 换 anthropic SDK + `/apps/anthropic` 端点。
  4. 探 GLM 模型名，确认 `glm-5.1`（带横杠）可用；`glm5.1` 报 InvalidParameter。
- **解决**：换 anthropic SDK，`base_url=.../apps/anthropic`，`model=glm-5.1`。
- **结果**：一次跑通，MyAgent 正常应答。
- **30 秒讲法**：调模型报 403，我先用错误码区分"key 错(401)还是权限错(403)"，再用多模型探测把"单模型"和"全账号"分开，最后从"另一个工具能用同一把 key"反推出是**端点/协议不匹配**——换对端点和 SDK 解决。定位靠分层排除，不是穷举瞎试。

### [2026-06-24] MyAgent「失忆」：原来 LLM 调用本身是无状态的
- **现象**：跟 MyAgent 说「我叫小明」，下一句问「我叫什么」，它完全不记得。
- **根因**：每次 `client.messages.create()` 都是一次**独立、无状态**的请求；模型不会自动记住上一轮。`ask()` 每次只把"当前这句"发过去，没带历史。
- **定位过程**：翻 `agent.py` 的 `ask()`——`messages` 数组每次只有当前 user 消息、没有累积历史 → 确认是"我们没喂历史"，不是模型坏了。
- **解决方向**（W3 要做的）：① **短期记忆** = 把对话历史一起塞进 `messages`（受上下文窗口限制）；② **长期记忆** = 把要点存向量库，需要时检索回来（RAG）。
- **30 秒讲法**：LLM API 是无状态的——「记忆」不是模型自带的，是工程上自己维护、再喂回去的。短期靠传 history（受上下文窗口约束），长期靠外部存储 + 检索。这也正是「上下文窗口」和「记忆」成为高频考点的原因。

### [2026-06-25] 加了字段却没同步 prompt：重试 3 次都救不回来
- **现象**：给 `AgentReply` 加了 `urgency` 字段，但 `SYSTEM` 没提它。运行后模型一直不输出 urgency，校验失败，**重试 3 次全挂**，模型死活不补。
- **根因**：**两个「真相源」脱节**——`SYSTEM`（告诉模型"字段只有 intent/reply/need_followup"）vs `AgentReply` 模型（校验时要 4 个字段）。模型按 SYSTEM 办事，和校验器**直接矛盾**；重试只是把同一个矛盾重复了 3 遍。（雪上加霜：重试话术把"缺字段"说成"无法解析、请返回合法 JSON"，可模型返回的本就是合法 JSON，更没法纠正。）
- **定位过程**：观察到 3 次错误一模一样、模型完全没尝试加 urgency → 意识到不是"模型笨"，是它收到的指令**自相矛盾**。
- **解决**：把 `SYSTEM` 改成 `+ json.dumps(AgentReply.model_json_schema())`——字段说明**从模型自动生成**。契约只剩一处（`AgentReply` 类），SYSTEM 自动带上 urgency → 模型第一次就给 → 一次过、零重试。
- **30 秒讲法**：重试 / 错误反馈**救不了"prompt 描述和 schema 从根上矛盾"**——打补丁打不过源头矛盾。真正的修法是**单一真相源**：让 prompt 从 schema 自动生成，二者永不脱节。我一开始靠重试兜底，后来定位到根因是契约漂移，改成单一真相源才彻底解决。
